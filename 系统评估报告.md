## 1. 架构评估

### 1.1 分层架构

```
┌─────────────────────────────────────────────┐
│  前端：单页 SPA (index.html)                  │
├─────────────────────────────────────────────┤
│  API 层：FastAPI + CORS + JWT                │
│    ├─ /api/chat        (非流式)              │
│    ├─ /api/chat/stream (SSE 流式)            │
│    └─ /api/auth/*      (注册/登录)           │
├─────────────────────────────────────────────┤
│  Agent 层：LangGraph + QwenChatModel         │
│    ├─ System Prompt (TCM_SYSTEM_PROMPT)      │
│    ├─ InMemorySaver (多轮对话记忆)            │
│    └─ 流式输出 (astream_events v2)            │
├──────────────────┬──────────────────────────┤
│  RAG 向量检索      │  GraphRAG 图检索          │
│  ├─ DashScopeEmb   │  ├─ Neo4jGraphStore      │
│  ├─ QdrantStore    │  ├─ GraphRetriever       │
│  └─ TCMKnowledge   │  └─ 图路径推理            │
│     SearchTool     │     (多跳关联发现)         │
├──────────────────┴──────────────────────────┤
│  Tools 层 (8 个 LangChain Tool)               │
│  精确查询: search_fangji / herb / acupoint   │
│           search_constitution                 │
│  语义检索: search_tcm_knowledge               │
│  图谱查询: search_symptom_path / relation     │
│           search_graph_entity                 │
├─────────────────────────────────────────────┤
│  Data 层：TCMDatabase (MySQL / SQLite 双模)    │
└─────────────────────────────────────────────┘
```

**评分：★★★★☆ (4/5)**

> 优点：层次清晰，职责分明。Agent ↔ Tools ↔ Data 的解耦做得不错，RAG 和 GraphRAG 两条检索链路并行设计合理。  
> 不足：全局 `agent_executor` 单例破坏了分层隔离；API 层绕过了 Agent 层的 `run_agent()` 函数，直接调用 `ainvoke`，造成逻辑分散。

### 1.2 技术栈

| 层级 | 技术 | 评价 |
|------|------|------|
| 大模型 | 通义千问 (DashScope) | 国产主流，合规性好 |
| Agent 框架 | LangGraph | 生态成熟，自带 checkpointer |
| API | FastAPI + SSE | 异步原生支持，流式友好 |
| 向量库 | Qdrant | 轻量，替代 Milvus 合理 |
| 图数据库 | Neo4j 5.26 | 知识图谱标准选型 |
| 关系数据库 | MySQL 8.0 / SQLite 回退 | 双模设计实用 |
| 包管理 | uv | 现代化，速度快 |
| 容器化 | Docker Compose | 部署标准化 |

**评分：★★★★★ (5/5)**

> 技术选型成熟且针对性强。Qdrant 替代 Milvus 的决策显著降低了部署复杂度（省去 etcd + MinIO）。

## 2. 请求处理流程

### 2.1 用户对话完整链路

```
用户输入「头痛发烧怎么办」
    │
    ▼
┌─ FastAPI /api/chat/stream ─────────────────┐
│ 1. JWT 认证 (get_current_user)              │
│ 2. session_id 生成                           │
│ 3. 调用 run_agent_stream()                   │
└──────────────────┬──────────────────────────┘
                   ▼
┌─ Agent 层 (LangGraph) ──────────────────────┐
│ 1. HumanMessage 入队                         │
│ 2. LLM 决策：该用什么工具？                    │
│    ├─ 已知方剂名 → search_fangji             │
│    ├─ 症状描述 → search_tcm_knowledge        │
│    └─ 多症状关联 → search_symptom_path        │
│ 3. 工具执行 → 结果返回                        │
│ 4. LLM 综合工具结果生成最终回复                │
│ 5. 对话状态存入 InMemorySaver (thread_id)     │
│                                                │
│ 工具调用示例：                                 │
│ 症状"头痛发烧"                                 │
│   ├─ search_symptom_path("头痛发烧")           │
│   │   → Neo4j 多跳路径查询                     │
│   │   → 发现: 头痛发烧→外感风寒→正柴胡饮颗粒    │
│   ├─ search_tcm_knowledge("头痛发烧外感")       │
│   │   → Qdrant 向量语义检索                    │
│   │   → 5 条相关方剂和药材                     │
│   └─ search_fangji("正柴胡饮颗粒")              │
│       → MySQL 精确查询                         │
│       → 方剂组成、功效、用法                    │
└──────────────────┬──────────────────────────┘
                   ▼
┌─ SSE 流式输出 ──────────────────────────────┐
│ event: token → 逐字输出                      │
│ event: tool_start → 前端显示「正在查方剂...」 │
│ event: tool_end → 前端显示工具结果            │
│ event: done → 附带 trace 摘要                │
└──────────────────┬──────────────────────────┘
                   ▼
              前端渲染回复
```

### 2.2 数据初始化流程

```
scripts/import_to_qdrant.py
    │
    ├─ 读取 data/herbs.jsonl (药材 ~3000 条)
    ├─ 读取 data/formulas_test.jsonl (方剂 ~300 条)
    ├─ 读取 data/diseases_test.jsonl (疾病 ~1000 条)
    ├─ 读取 data/syndromes_test.jsonl (证型 ~500 条)
    │
    ├─ DashScopeEmbedding.encode() → 768 维向量
    └─ QdrantStore.insert() → Qdrant collection

src/graphrag/seed_graph.py
    │
    └─ Neo4j 写入实体 + 关系 (药材→方剂→症状→经络)
```

**评分：★★★★☆ (4/5)**

> 流程设计完整，SSE 事件模型让前端能展示工具调用过程，用户体验好。  
> 问题：`InMemorySaver` 在服务重启后所有对话历史丢失，用户会感到「更短的记忆」。

## 3. 模块设计评估

### 3.1 Agent 层 (`src/agent/`)

| 维度 | 评价 |
|------|------|
| `QwenChatModel` 自实现 | 完整支持 Tool Calling + 流式，适配质量高 |
| System Prompt | 结构清晰，工具选择策略写得很好，场景覆盖全 |
| `run_agent_stream()` 追踪 | 手动计时 + logger 方式实用，但代码侵入性强 |
| `TracingCallback` | 已定义但从未被使用——死代码 |

**问题**：
- LLM 层与 DashScope SDK 强耦合，如果要换模型（如千问 2、GLM-4）需要重写 `QwenChatModel`
- `bind_tools()` 使用 `model_copy()` 可能存在浅拷贝陷阱
- 日志 handler 在模块 import 时注册，多次 import 可能导致重复输出

### 3.2 API 层 (`src/api/`)

| 维度 | 评价 |
|------|------|
| FastAPI + 异步 | 选型正确，原生支持流式 |
| JWT 认证 | 基本可用，PBKDF2 密码哈希 |
| 前端内嵌 | 单文件 SPA，部署方便 |
| CORS | `allow_origins=["*"]` 过于宽松 |

**问题**：
- `sys.stdout` 全局重定向（第 11 行），影响面太广
- `/api/chat` 中绕过 `run_agent()` 直接调 `ainvoke`，导致 trace 日志缺失
- SSE `event: error` 行后面缺少 `\n`（第 148 行），可能导致客户端解析异常
- 认证模块 `_init_user_table()` 在 import 时自动执行，模块加载顺序不可控

### 3.3 工具层 (`src/tools/`)

| 维度 | 评价 |
|------|------|
| 工具设计 | 8 个工具覆盖精确/语义/图谱三种检索模式 |
| 懒初始化 | `_get_retriever()` 避免启动时强制连接 Neo4j |
| Pydantic 输入模型 | 类型安全，Field description 为 LLM 提供好线索 |

**问题**：
- 每个 Tool 的 `db` 属性用 `default_factory=lambda: TCMDatabase()` 创建独立连接，同一轮对话多个工具调用会创建多个连接
- 工具之间没有共享上下文（如先查图谱获得方剂名，再查 DB 需要两次独立调用）
- Neo4j 不可用时 GraphRAG 工具会直接崩溃，无降级处理

### 3.4 数据层 (`src/data/`)

| 维度 | 评价 |
|------|------|
| MySQL/SQLite 双模 | 实用设计，SQLite 适合开发/测试 |
| 参数化查询 | 用 `_ph` 占位符，防注入 |

**问题**：
- 每次查询创建新连接，无连接池复用
- `_exec_sql` 返回类型不一致（`list[dict]` 或 `None`），调用方需要判空
- `src/data/database.py` 文件约 800 行，单文件包含了所有 DB 逻辑，建议拆分

### 3.5 RAG 层 (`src/rag/`)

| 维度 | 评价 |
|------|------|
| Qdrant 替代 Milvus | 部署从 5 容器降到 1 容器，优秀决策 |
| Embedding 维度 | 768 维硬编码，若换模型不匹配 |

**问题**：
- `QdrantStore.connect()` 独立调用，若忘记调用则后续操作静默失败
- 没有实现增量更新或去重逻辑，`--reset` 是全量重建

### 3.6 GraphRAG 层 (`src/graphrag/`)

| 维度 | 评价 |
|------|------|
| 多跳路径推理 | 通过 Cypher 查询实现症状→方剂→药材的关联发现 |
| TXT 解析器 | 从非结构化文本提取结构化知识，解决数据源问题 |

**问题**：
- 8 个 `.txt` 源文件耦合在代码仓库中，数据更新需要重新构建
- 图数据库写入脚本 `seed_graph.py` 依赖 `.jsonl` 中间产物，链路长

**评分：★★★☆☆ (3/5)**

> 模块职责分得开，但内部耦合度高（单文件过大、连接管理分散）。连接池、懒加载、错误降级是明显的改进方向。

## 4. 安全性评估

### 4.1 发现的隐患

| 严重度 | 问题 | 位置 |
|--------|------|------|
| 🔴 高 | API Key 明文存储于 `.env` | `.env` 文件 |
| 🔴 高 | CORS 全开 `allow_origins=["*"]` | `server.py:75` |
| 🟡 中 | JWT SECRET_KEY 回退到 API Key | `auth.py:19` |
| 🟡 中 | 无请求频率限制 | `server.py` 全局 |
| 🟡 中 | SQL f-string 拼接表名（不可注入但不够规范） | `auth.py:69` |
| 🟢 低 | 密码最小长度仅 6 位 | `auth.py:136` |
| 🟢 低 | Token 无刷新机制 | `auth.py:22` |

### 4.2 建议

```
紧急：
  1. 轮换已暴露的 API Key
  2. 限制 CORS 为前端域名
  3. JWT SECRET_KEY 使用独立随机字符串

重要：
  4. 添加 slowapi 做频率限制
  5. 密码最小长度提升到 8 位 + 复杂度检查
```

**评分：★★☆☆☆ (2/5)**

> 安全性是当前最薄弱的环节。CORS 全开 + API Key 明文 + 无限流，在生产环境不可接受。

## 5. 可靠性评估

### 5.1 异常处理模式

```python
# 典型模式（存在项目中多处）：
try:
    result = await agent_executor.ainvoke(...)
except Exception as e:
    raise HTTPException(status_code=500, detail=f"Agent 执行错误: {str(e)}")
```

- `except Exception` 太宽泛，会吞掉 `KeyboardInterrupt` 和 `SystemExit`
- 没有对可恢复错误（超时、限流、网络闪断）做重试
- Neo4j 连接失败时 GraphRAG 工具直接抛异常，不会降级到纯 RAG 模式

### 5.2 已知脆弱点

| 场景 | 行为 | 风险 |
|------|------|------|
| Qdrant 未启动 | 语义检索工具报错 | Agent 缺少关键知识库 |
| Neo4j 未启动 | 图谱工具报错 | 关系推理不可用 |
| 通义千问 API 限流 | 无重试，直接返回 500 | 用户体验差 |
| 服务重启 | InMemorySaver 清空，所有对话历史丢失 | 用户困惑 |
| 并发请求同一 session | InMemorySaver 无锁保护 | 数据竞争 |

**评分：★★★☆☆ (3/5)**

> 对依赖服务的故障容忍度低。缺少 circuit breaker 或 graceful degradation 模式。

## 6. 性能评估

### 6.1 关键路径耗时（估算）

| 阶段 | 耗时 | 备注 |
|------|------|------|
| JWT 验证 | < 1ms | 纯计算 |
| LLM 首次决策 | 1-3s | 含工具选择推理 |
| Qdrant 向量检索 | 50-200ms | 取决于数据量 |
| Neo4j 图查询 | 100-500ms | 取决于跳数 |
| MySQL 精确查询 | 5-20ms | 简单查询 |
| LLM 生成回复 | 2-8s | 含流式输出 |
| **整轮对话** | **5-15s** | 取决于工具调用次数 |

### 6.2 瓶颈分析

1. **数据库连接**：每次工具调用创建新连接，高频对话时开销累积
2. **无缓存**：常见药材/方剂的查询结果每次都会重新查 DB
3. **Embedding 维度硬编码 768**：实际 DashScope embedding 可能是 1024/1536 维，不匹配会降低检索质量
4. **串行工具调用**：Agent 一次只调一个工具，复杂问诊需要多轮串行调用

**评分：★★★☆☆ (3/5)**

> 单用户场景足够，多用户并发时连接创建和序列化开销会暴露。

## 8. 代码质量评估

### 8.1 正面

- 类型注解使用广泛（`Typing` + Pydantic）
- 函数粒度适中，单函数一般不超过 50 行
- 变量命名清晰，`search_fangji` / `search_symptom_path` 等自解释
- System Prompt 是一大亮点，工具选择策略写得精准

### 8.2 负面

| 问题 | 示例 | 影响 |
|------|------|------|
| **模块级副作用** | `core.py` 日志配置、`auth.py` 建表、`server.py` stdout 重定向 | 不可预测的行为 |
| **死代码** | `TracingCallback` 类定义了但从未使用 | 维护困惑 |
| **全局可变状态** | `agent_executor`、`_graph_store`、`_graph_retriever` | 测试困难，并发风险 |
| **连接泄漏风险** | 每次工具调用 new `TCMDatabase()`，无显式 close 保证 | 长时间运行可能耗尽连接 |
| **硬编码值** | embedding 维度 768、token 有效期 24h、日志截断 80/200 字符 | 缺少灵活性 |
| **超大单文件** | `database.py` 约 800 行 | 难以维护 |

**评分：★★★☆☆ (3/5)**

> 基本打得标准，但模块级副作用和全局状态是需要优先清理的技术债。
