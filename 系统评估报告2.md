## 二、架构分析

### 2.1 整体分层

```
┌──────────────────────────────────────┐
│         React 19 Frontend            │  ← Public / Professional / Admin 三端
├──────────────────────────────────────┤
│      FastAPI REST + SSE Stream       │  ← 认证、Chat、Conversation、Model Config
├──────────────────────────────────────┤
│   LangGraph Multi-Agent Supervisor   │  ← Supervisor → 5 个专业 Agent
├──────────────────┬───────────────────┤
│   Middleware Chain│  Memory System   │  ← Guardrails, PII, Logging, Context
├──────────────────┼───────────────────┤
│   MCP Tools × 8  │  LLM Provider     │  ← 中药材/方剂/古籍/医案搜索
├──────────────────┼───────────────────┤
│  PostgreSQL      │  Neo4j            │  ← 22 张业务表 + 知识图谱
│  Qdrant          │  Redis            │  ← 向量检索 + 缓存
└──────────────────┴───────────────────┘
```

### 2.2 Agent 编排 — Supervisor 模式（9.0/10）

采用 LangGraph 的 Supervisor 模式，包含 **5 个专业 Agent**：

| Agent | 职责 | 激活条件 |
|-------|------|---------|
| `SUPERVISOR` | 主协调器，任务分解 + 结果合成 | 所有请求 |
| `DIAGNOSIS` | 辨证问诊（十问歌流程） | 症状描述 |
| `PRESCRIPTION` | 方剂推荐 + 十八反十九畏校验 | 方剂需求 |
| `WELLNESS` | 养生建议 + 体质调理 | 养生问题 |
| `HERB` | 药材查询 + 配伍检查 | 药材问题 |
| `GENERAL` | 中医基础科普、闲聊 | 一般性问题 |

**关键设计亮点：**

- **Direct-Pass 机制**：子 Agent 可直接响应用户，避免 Supervisor 每次都汇总再输出，减少延迟
- **4 种响应类型**：`SYNTHESIZED`（综合）/ `DIRECT_PASS`（直传）/ `PARTIAL`（部分）/ `HANDOFF`（移交）
- **并行控制**：`asyncio.Semaphore` 限制最大并行 Agent 数 = 3
- **三级复杂度**：诊断子图按 simple / moderate / complex 分流，complex 走 DeepSearch Agent（5 个专家子 Agent 并行分析）

**可改进**：缺少 Swarm/协作模式，Agent 之间不能动态协商投票。

### 2.3 请求处理全链路

```
用户输入
  → Controller (chat_controller.py)
  → Service (chat_servcie.py)
  → 解析 LLM 配置（数据库查询供应商/模型/Key）
  → LangGraph 图启动
  → Middleware Chain (before)
    ├── Guardrails: L1 规则 + L3 LLM 双重安全检测
    ├── PII: 脱敏处理
    ├── Logging: 请求日志
    ├── Context: 上下文管理
    ├── Filesystem: 文件系统访问
    ├── Memory: 记忆注入
    └── Focus: 焦点管理
  → Intent Recognition: L1 规则层 + L3 LLM 层
  → 条件路由到 5 个子图
  → Middleware Chain (after)
  → SSE 流式输出到前端
```

### 2.4 诊断子图深度拆解

```
initial → cold_heat → sweat → stool_urine → diet → sleep → tongue → pulse → complete
```

完全遵循中医"十问歌"传统问诊流程，每一步收集特定信息，逐步缩小辨证范围。Complex 病例触发 **DeepSearch Agent**（5 个专家子 Agent 并行分析各维度后再汇总）。

## 三、各维度评分

### 3.1 架构设计 9.0/10

| 评价项 | 得分 | 说明 |
|--------|:--:|------|
| 分层清晰 | 9 | Controller → Service → Agent → Middleware → Data，标准工业级分层 |
| 多 Agent 编排 | 9 | Supervisor 模式完整，含 Direct-Pass 优化 |
| 子图设计 | 9 | 5 个意图子图 + 诊断三级复杂度分流 |
| 状态管理 | 9 | Pydantic BaseModel + Annotated reduce 函数，类型安全 |
| 中间件链 | 9 | 7 个中间件可插拔组合，基类抽象清晰 |

**亮点**：LangGraph 图结构 + 中间件链的架构设计与 Spring Boot 的 Filter Chain 异曲同工，是生产级 Agent 系统的范式级实现。

### 3.2 安全与护栏 9.5/10

| 评价项 | 得分 | 说明 |
|--------|:--:|------|
| Guardrails 双层检测 | 10 | L1 规则（关键词+正则，毫秒级）+ L3 LLM（语义级兜底） |
| LLMbda 形式化验证 | 10 | 对 LLM 输出做形式化约束验证，行业前沿 |
| JWT + 刷新令牌 | 9 | 三角色（Public/Professional/Admin）认证隔离 |
| PII 脱敏 | 9 | 中间件自动检测和脱敏处理 |
| 紧急情况检测 | 9 | 关键词+规则匹配，自动建议就医 |
| Auth 中间件 | 8 | 请求级身份验证 |

**亮点**：LLMbda 形式化验证 + 双层 Guardrails 的组合是业界极少见的完整安全链路。PII 脱敏和紧急情况检测进一步强化了医疗场景的安全保障。

### 3.3 数据层设计 9.0/10

| 评价项 | 得分 | 说明 |
|--------|:--:|------|
| PostgreSQL 表设计 | 9 | 22 张表，覆盖账户/患者/医生/病例/药材/方剂/证型/古籍/供应商/模型 |
| 知识图谱 | 8 | Neo4j 存储药材-症状-证型-方剂关系，含 Cypher 模板库 |
| 向量检索 | 8 | Qdrant 支持语义检索 |
| Redis 缓存 | 8 | 缓存 + Session 管理 |
| SQLModel ORM | 8 | 异步 + 同步双 Session 管理 |

**亮点**：22 张表的完整业务建模 —— 从账户到药材库存到舌诊分析到 LLM 供应商配置全部入表，并且 init.sql 自动生成机制保证了 Schema 可复现。

**不足**：Cypher 模板局限于预定义查询，缺乏基于 LLM 的动态 Cypher 生成。

### 3.4 MCP 工具协议 8.0/10

| 评价项 | 得分 | 说明 |
|--------|:--:|------|
| MCP Server | 9 | 8 个 TCM 工具，语义完整的输入输出 schema |
| MCP Client | 9 | stdio + SSE 双传输模式 |
| LangChain 适配 | 8 | 自定义 MCPToolkit + 动态 Pydantic 生成 |
| 真实数据连接 | 2 | **致命伤** — 工具返回内置 mock 数据，未接 Neo4j/Qdrant |

**亮点**：MCP 协议在中医领域最完整的实现之一，双传输模式远超多数项目。

**致命伤**：有插座没通电 —— 8 个工具全部返回硬编码 mock 数据，等于整个工具层是空壳。

### 3.5 记忆系统 8.5/10

三层认知记忆架构：

| 层级 | 名称 | 存储 | 用途 | 有效期 |
|------|------|------|------|--------|
| L1 | 工作记忆 | LangGraph Checkpointer (PostgresSaver) | 当前对话上下文 | 会话期间 |
| L2 | 情景记忆 | Mem0 + Qdrant | 跨会话用户画像、症状历史 | 长期 |
| L3 | 长期记忆 | PostgreSQL | 诊断历史、体质档案、处方记录 | 永久 |

**亮点**：三层记忆模型实现了从"瞬时"到"永恒"的认知跨度，TCMMemoryMetadata 包含症状/证型/方剂/季节等中医特有字段，时间衰减和冲突检测进一步提升了记忆质量。

**不足**：子图间记忆透传不完整，DeepSearch Agent 的部分子 Agent 无法访问完整的用户历史。

## 七、总结

**这是一套在 Agent 架构设计上达到业界前沿水平的中医问诊系统。**

架构层面，Supervisor 多智能体编排 + 中间件链 + LLMbda 形式化验证 + 三层记忆模型的组合足以作为 Agent 系统的教学范例。22 张表的完整业务建模、MCP 全栈实现、三端分离的前端设计，都展现了扎实的工程素养。

然而，**MCP 工具全 mock 数据**和**零可观测性**这两个短板，使得这套架构优美的系统在"真正能用"这个标准上大打折扣。如果你把它当作一个 Agent 架构的参考实现，它值 9 分；如果你要直接部署给患者使用，实际可用性可能只有 6 分。

当前状态最适合作为：**Agent 架构的研究原型 + 教学演示**。要走向生产，必须优先打通数据链路和补齐可观测性。
